3. Тестирование решения
   3.1. Подход к проверке итогового продукта
        3.1.1. Условия применимости решения
               — соответствие функциональным и нефункциональным требованиям
               — достижение согласованного состояния на уровне бизнес-логики
                 как итогового оракула корректности
        3.1.2. Выбранные подходы к тестированию
               — unit-тестирование логики библиотеки
               — сквозное тестирование на наборе специально созданных сервисов,
                 имитирующих системный дизайн и инфраструктуру реальных
                 микросервисных приложений
        3.1.3. Соответствие подходов требованиям
               — какие требования закрываются unit-тестами
               — какие требования закрываются сквозным тестированием
   3.2. Unit-тестирование логики библиотеки
        3.2.1. Объекты покрытия
               — модули инфраструктурной надёжности: backoff, retry
               — внутренние компоненты координации: executor, inbox,
                 outbox (writer)
               — высокоуровневые абстракции Step и Controller покрываются
                 сквозным сценарием на демонстрационной системе
        3.2.2. Проверка позитивных и негативных сценариев
               на уровне бизнес-логики покрываемых модулей
        3.2.3. Оценка покрытия и критерии достаточности
   3.3. Применение библиотеки разработчиком
        3.3.1. Описание типового сценария интеграции библиотеки в сервис
        3.3.2. Определение шагов саги
               — реализация прямой операции (Execute)
               — реализация компенсирующей операции (Compensate)
               — формирование и публикация сообщения-результата шага
        3.3.3. Конфигурация саги на стороне координатора
               — декларативное описание последовательности шагов
               — настройка маршрутизации сообщений (RoutingConfig)
               — параметры повторов, таймаутов и стратегий обработки ошибок
        3.3.4. Подключение адаптера брокера сообщений
               — готовые адаптеры библиотеки: NATS (pubsub/nats),
                 Kafka-адаптер на стороне демо-сервисов
               — возможность реализовать собственный адаптер через интерфейс брокера
        3.3.5. Обеспечение атомарности на стороне сервиса
               — применение Transactional Outbox для публикации событий
               — применение Inbox для идемпотентной обработки входящих сообщений
               — требования к схеме БД сервиса
        3.3.6. Создание и запуск контроллера саги
               — инициализация зависимостей и регистрация шагов
               — запуск воркеров Outbox/Inbox и слушателей брокера
        3.3.7. Подключение наблюдаемости
               — экспорт метрик через Prometheus-совместимый /metrics сервиса
               — настройка распределённого трейсинга OpenTelemetry (OTLP)
   3.4. Тестовая система для сквозного тестирования
        3.4.1. Обоснование необходимости демонстрационной системы
               — недостаточность unit-тестов для проверки распределённого сценария
               — необходимость наблюдения за сквозным поведением системы
                 и имитации инфраструктуры реальных микросервисных приложений
        3.4.2. Обоснование выбора предметной области
               — типичность сценария для реальных систем и распространённость
                 задачи распределённого бронирования в e-commerce
        3.4.3. Описание предметной области (сага бронирования путешествия)
        3.4.4. Состав сервисов, их API и схемы баз данных
               — booking, flights, hotels; swagger-документация и proto-контракты;
                 миграции БД для каждого сервиса
        3.4.5. Схема взаимодействия сервисов
        3.4.6. Обоснование архитектуры демонстрационной системы
               — выбор хореографической координации с финализирующим шагом
                 на стороне booking-сервиса
               — обоснование выбора инструментов (Kafka как брокер,
                 Postgres и MySQL как разные СУБД участников саги)
                 и их соответствие реальным production-окружениям
        3.4.7. Инфраструктура системы (docker-compose)
               — брокер сообщений: Kafka (Confluent) + Zookeeper + kafka-ui
               — СУБД участников: PostgreSQL (booking, flights), MySQL (hotels)
               — распределённый трейсинг: Jaeger (приём по OTLP)
               — сбор метрик: Prometheus-совместимый эндпойнт /metrics
                 на каждом сервисе (инспекция вручную; Prometheus-сервер
                 в docker-compose не разворачивается)
   3.5. Сквозное (E2E) тестирование
        3.5.1. Сценарии тестирования
               — Happy path: успешное прохождение саги
               — Negative path: компенсация при сбое шага
        3.5.2. Автоматически проверяемые аспекты
               — корректность доставки сообщений между сервисами
                 (косвенно через итоговое состояние БД)
               — согласованность бизнес-данных в БД трёх сервисов
               — работоспособность саги с разными СУБД в рамках одного сценария
                 (PostgreSQL + MySQL)
        3.5.3. Аспекты, проверяемые наблюдением, а не автотестами
               — сбор метрик через /metrics сервисов
               — журналирование хода выполнения саги
               — распределённый трейсинг в Jaeger UI
               — поддержка разных брокеров подтверждается наличием готового
                 NATS-адаптера в библиотеке и его использованием в примерах
        3.5.4. Критерии завершённости теста и оракул проверки
               — согласованное состояние бизнес-данных в БД всех сервисов
                 как основной оракул
               — ручное тестирование и проверка по метрикам и трейсам сервисов
               — автоматизированный запуск тестов и проверка транзакций в БД
